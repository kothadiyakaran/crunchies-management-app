# Sprints 9 + 10 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close Phase 1 build. Sprint 9 wires the last spec-required surfaces (Settings + event uplift) and runs a polish/accessibility/performance/backfill pass; Sprint 10 verifies the full §3 eight-flow surface end-to-end and resolves any debt. Single review checkpoint after both sprints.

**Architecture:**
- Settings as a single-row table read once per session into a React context (mirrors `BUSINESS_INFO` shape — drop-in replacement).
- Event uplift in `production/algorithm.ts` as a new input field `eventUplift: Record<string, number>` with per-product source labels for the subtitle. `getProductionThisWeek` computes the uplift client-side from `event_demand` rows of events touching W (same pattern as committed_demand fetch).
- Backfill is a Node TS script driven by a CSV, idempotent (lookup-by-phone for customers, composite key for orders), with `--dry-run` default.

**Tech Stack:** Vite + React 18 + TS strict + Supabase JS + Vitest/RTL + Playwright. No new deps.

### T9.1 — Settings table migration + API

**Files:**

- Create: `src/features/settings/api.ts`
- Create: `src/features/settings/api.test.ts`

**Schema:** single-row table `business_settings` with `id uuid primary key default gen_random_uuid()`, columns mirroring `BusinessInfo` (name text not null, tagline text, address_lines text[] not null default '{}', gst_line text, phone text, whatsapp text, email text, bill_footer text not null default 'Thank you', signature_line text not null default '— Archana'), plus `updated_at timestamptz default now()`. RLS: enable, `authed_all` policy for SELECT/UPDATE for authenticated; **anon SELECT allowed on a public column subset only** (name, tagline, whatsapp — the three the public form/confirmation page need). Seed one row matching `src/lib/business.ts:BUSINESS_INFO` current values.

For the anon-readable subset: rather than risk RLS column-level complexity, expose **one SECURITY DEFINER RPC** `public_get_business_identity()` returning `{ name, tagline, whatsapp }` — anon retains zero direct table access (consistent with Sprint 7 ADRs).

**API:**
- `getSettings(): Promise<BusinessInfo>` — authenticated read, returns the single row mapped to `BusinessInfo`
- `updateSettings(patch: Partial<BusinessInfo>): Promise<BusinessInfo>` — updates the single row, returns updated mapping
- `getPublicBusinessIdentity(): Promise<{ name: string; tagline: string | null; whatsapp: string | null }>` — calls the RPC

**Context:** `SettingsProvider` fetches once on mount, exposes `useSettings()` hook returning `{ settings: BusinessInfo, refresh: () => Promise<void>, loading: boolean }`. Provider mounts inside `<Protected />` so authenticated routes have access. Public routes use the RPC directly via `getPublicBusinessIdentity()` (no context).

**Tests:** 4 unit tests on api.ts shape mapping (snake_case columns ↔ camelCase BusinessInfo, null handling on optional fields).

### T9.8 — Backfill script

**Files:**
- Create: `scripts/backfill-notebook.ts`
- Create: `scripts/backfill-notebook.README.md`
- Create: `scripts/backfill-notebook.test.ts` (Vitest)

**CSV schema (input):**
```
customer_name,customer_phone,customer_channel,ordered_on,product_name,qty,unit_price,payment_status,target_fulfilment_date,notes
"Sunita Patil","9876543210","Personal","2024-12-10","Laddu box",2,200,"paid","2024-12-12",""
```

**Behaviour:**
- Reads CSV from `--input <path>` (required)
- Default mode is `--dry-run` (no writes; prints a plan summary: "Would create N customers, M orders, K order_items")
- Pass `--apply` to actually write
- Idempotent:
  - Customer: lookup by `phone` (with `cleanPhone()`); skip create if exists, reuse id
  - Product: lookup by `name` (case-insensitive); error out if missing — user must create products via UI first (the backfill won't invent products)
  - Order: composite key (customer_id, ordered_on, hash of items) — skip if a matching order exists
- Logs every action to stdout in the form `[customer] CREATE Sunita Patil` / `[customer] EXISTS Sunita Patil` / `[order] CREATE …` etc.

**Tests:** unit tests against a mocked supabase client — verify dry-run writes nothing, verify dedup logic for customer+order, verify product-missing aborts the row with a clear error.
